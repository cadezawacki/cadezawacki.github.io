Room Manager — Folder / Tag Prototypes

Four different ways to organize rooms. Each option shows the menubar (loaded into a "work" context) and the Room Manager modal. Click through to feel the interaction; the data model is shared between variants so the same set of rooms appears everywhere.

Variant A

Nested Folders
Full tree like Finder. Folders contain folders contain rooms. Breadcrumb lets you "enter" a folder as your active context.

Variant B

Workspaces (Flat)
One level deep. Each room belongs to one workspace. Pick a workspace and the tabs filter to it. Simple, fast to grasp.

Variant C

Tags (Multi-belong)
Rooms have one or more tags. Tag chip filters the tab strip. Same room can live in multiple "contexts" at once.

Variant D

Path Prefixes
Folders are implicit in slash-delimited names: work/notes. Lightest implementation; folders emerge from naming.
A · Nested Folders (Finder-style)

Rooms live inside folders, folders can nest. A breadcrumb pill in the menubar shows your current folder context. Tabs only show siblings of the active room (children of the current folder). Click the breadcrumb to jump anywhere in the tree — including "All Rooms" at the root.

Pros

  • Matches your "load into a project" mental model exactly
  • Scales infinitely; deep nesting works
  • Familiar (every file system uses this)
  • Breadcrumb gives you "where am I" at a glance

Cons

  • Most code to write (tree ops, expand/collapse, drag-into-folder)
  • Room can only live in one folder
  • Breadcrumb gets long on small screens

B · Workspaces (Flat, one level)

One level of grouping — call them "workspaces" or "projects". Each room belongs to exactly one workspace. A workspace pill in the menubar acts as the context switcher; tabs show only rooms in the active workspace. No nesting, no breadcrumbs — fastest to ship and reason about.

Pros

  • Simplest model — easy to migrate existing rooms
  • Pill + tabs read instantly: "I'm in <ws>, these are its rooms"
  • No tree UI needed — flat list inside each workspace
  • Most apps you've used (Slack, Notion teamspaces) work this way

Cons

  • No sub-projects — flat at one level
  • Heavy users may want hierarchy later
  • Room can only live in one workspace

C · Tags (Flexible, multi-belong)

No folders — rooms carry one or more tags. The menubar shows tag chips; click one to filter the tab strip to rooms with that tag. Same room can appear under multiple tags (e.g. #work AND #urgent). Most flexible, least "container-like".

Pros

  • Rooms can live in multiple "places" at once
  • Easy cross-cutting filters (e.g. show all #urgent across projects)
  • No tree to maintain; tags self-organize
  • Lightweight migration: untagged rooms work as today

Cons

  • Doesn't match the "load into a project" framing as crisply
  • Tag sprawl over time; users need to curate
  • Less visual hierarchy at a glance

D · Path Prefixes (Implicit folders)

Folders are baked into the name itself: work/notes, work/client-x/specs. The app auto-groups by prefix in the manager and offers a prefix filter pill in the menubar. Zero new schema — the only change is how names are parsed and displayed.

Pros

  • Minimal code change — same room model as today
  • Sync/Firebase path layout already mirrors this
  • Power users can type work/x/notes directly
  • Easy to rename a "folder" — string replace prefix

Cons

  • Slashes in names feel un-pretty next to bare rooms
  • Renaming is destructive (changes Firebase room path)
  • No "move to folder" — it's literally a rename
  • Easy to mess up by typo (work/ vs Work/)

My recommendation

Given the framing — "load into a project, dropdown filters to that folder, jump back to parent" — Variant A (Nested Folders) matches the mental model most precisely. But it has the most implementation surface area.

If you want to ship something solid in one pass and only add hierarchy later if you actually miss it, Variant B (Workspaces) gets you 80% of the value at ~30% of the code, and stays out of the way on mobile.

Variant C (Tags) is the right call if you find yourself wanting the same room to live in multiple contexts (e.g. a meeting that's both #work and #urgent), but it fits the "project" framing less directly.

Variant D (Path Prefixes) is the cheapest. Worth considering if the data model stability matters more than the UX polish, since it doesn't touch Firebase at all.
