§1 · Open questions, answered

Can a doc live in multiple workspaces?
No — strict one room → one workspace.
Keeps "delete" unambiguous and counts honest. If you ever need cross-listing, we layer tags in as a complementary concept on top of workspaces — but don't bake it in now. (Note: share-link already covers "give someone this room without restructuring my org.")
Where do new rooms go when I'm in "All Rooms"?
Into "Unfiled" — a real, auto-managed workspace.
"All Rooms" is a view, not a workspace. Every room must belong to exactly one workspace; rooms with no chosen home go to Unfiled. Migration path is trivial: every existing room lands here on first load.
How do I move a room between workspaces?
Three converging paths: drag, ⚙ menu, right-click.
All call the same setWorkspace(roomId, ws). Drag-onto-pill for desktop power users; ⚙ → "Move to…" for mobile + discovery; right-click on room tabs for desktop. Pick whichever feels natural.
Custom workspace colors — helpful or hurtful?
Helpful, with guardrails: 8 curated colors, not free-form.
Color shows on the pill and on "All Rooms" badges — not on every room tab (would be visual noise while typing). Limited palette = no clashes, consistent accessibility, painless theming. Default is neutral grey.

§9 · Sort order

Two layers: workspace order (in the switcher), and room order (within a workspace).

Workspace order
  • Drag in the "Workspaces" tab of Room Manager. Persists to workspaces[].
  • Unfiled is pinned last — can't be reordered.
  • "All Rooms" is pinned first in the switcher dropdown (above the divider).
Room order (per workspace)
  • Manual (default) — drag in Room Manager; persists per workspace.
  • Auto-sort options in the workspace's ✎ settings: manual, A-Z, recent edit, created.
  • Tab strip respects the same order — no second source of truth.

§10 · Data model & migration

Minimal-touch schema. Lives entirely in localStorage; Firebase is unchanged.

Schema (TypeScript-ish)
// localStorage: 'cade-workspaces' (new) type Workspace = { id: string, // e.g. "ws_3a9f" — generated, immutable name: string, // user-facing, editable color: "neutral" | "teal" | "blue" | "green" | "amber" | "orange" | "pink" | "purple", sort: "manual" | "az" | "recent" | "created", }; // localStorage: 'cade-rooms' (existing — extended) type Room = { name: string, workspaceId: string, // NEW — points to Workspace.id, or "unfiled" order: number, // NEW — manual sort within workspace }; // localStorage: 'cade-active-workspace' (new) type ActiveContext = string | "all"; // workspace id, or "all" for All Rooms view
Migration on first load with the new build
  • If cade-workspaces doesn't exist — create it as []. No "default" workspace forced on the user.
  • Every existing room gets workspaceId: "unfiled" and keeps its position via the existing index → order.
  • Active context = the workspace containing the previously-active room, or "all" if it's in Unfiled.
  • No Firebase touch. Workspaces are purely a client-side organization layer; the room's Firebase path is its name, same as today.

→ A user who already has 5 rooms opens the new build and sees: pill = "All Rooms", tabs = their 5 rooms (each with a neutral grey dot), Room Manager has one workspace ("Unfiled") containing all 5. They can ignore workspaces entirely and the app keeps working.

Sync implications: Workspaces are local. Different devices may organize the same set of rooms differently. If you want workspaces to sync too, that's a separate feature (encrypted blob written to config/{deviceKey}/workspaces in Firebase). Mention this if you want it built in v1.

§11 · Things still TBD — your call

A few I deliberately deferred. Pick before I start implementing.

  1. Sync workspaces across devices? Default: no (local-only). Yes adds an encrypted config blob to Firebase.
  2. Workspace icons in addition to color? Default: no (color + name is enough). Could add a small emoji slot if you want stronger visual identity.
  3. Per-workspace room cap, or one global cap? Default: global cap (today's behavior). Per-workspace would mean a small caps editor in the Workspaces tab.
  4. Command palette (⌘K)? Default: yes — same picker as the pill dropdown, with fuzzy search across both workspaces and rooms.
  5. "Pin to multiple workspaces" as a future tag-like feature? Default: not in v1. Reopens "delete vs unpin" ambiguity; ship strict first.
  6. Empty-workspace UX. When a workspace has 0 rooms: show a soft empty state with "+ Add room" CTA, or hide the workspace from the switcher? I'd default to show (predictable).
